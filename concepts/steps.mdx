## Overview

Steps are the building blocks of workflows. Each step represents a distinct stage in a business process, with specific behavior, assignment rules, and transition logic.

## Step Types

The platform supports six core step types:

```python
StepType = Literal[
    "start",        # Entry point (exactly one required)
    "end",          # Terminal step
    "human_task",   # Requires user action
    "system_task",  # Automated processing
    "subworkflow",  # Launches child workflow
    "decision"      # Conditional routing
]
```

### Step Type Behavior

<AccordionGroup>
  <Accordion title="start" icon="play">
    The entry point of every workflow. Validation ensures exactly one start step exists. The start step is automatically entered when a workflow instance begins.
    
    **Validation**: `workflows.py:79-81`
    ```python
    start_steps = [step.stepCode for step in payload.steps if step.stepType == "start"]
    if len(start_steps) != 1:
        errors.append("Exactly one start step is required.")
    ```
  </Accordion>
  
  <Accordion title="end" icon="flag-checkered">
    Terminal steps that complete the workflow. When an end step is entered, the workflow status transitions to `completed` or `rejected` based on the triggering action.
    
    **Implementation**: `runtime.py:885-887`
    ```python
    if step_definition["step_type"] in {"end"} or step_definition["is_terminal"]:
        _complete_workflow(cursor, workflow_instance, step_instance, trigger_action)
        return {"step_instance_id": step_instance["id"], "next_step_code": None}
    ```
  </Accordion>
  
  <Accordion title="human_task" icon="user">
    Steps requiring human intervention. The workflow pauses and creates actionable tasks for assigned users. Supports multiple approval modes and escalation policies.
    
    **Implementation**: `runtime.py:889-894`
    ```python
    if step_definition["step_type"] == "human_task":
        _create_human_tasks(cursor, workflow_instance, step_instance, step_definition)
        return {
            "step_instance_id": step_instance["id"],
            "next_step_code": step_definition["step_code"],
        }
    ```
  </Accordion>
  
  <Accordion title="system_task" icon="robot">
    Automated steps that execute immediately without user intervention. Used for data transformation, API calls, or business logic evaluation.
  </Accordion>
  
  <Accordion title="subworkflow" icon="diagram-project">
    Steps that launch a child workflow and pause the parent until the child completes. See [Subworkflows](/concepts/subworkflows) for details.
    
    **Implementation**: `runtime.py:896-931`
  </Accordion>
  
  <Accordion title="decision" icon="code-branch">
    Conditional routing steps that evaluate expressions or conditions to determine the next path.
  </Accordion>
</AccordionGroup>

### Revert Control

<ResponseField name="allowRevert" type="boolean" default="true">
  Whether users can revert the workflow back to this step
</ResponseField>

```python
# Revert transitions are filtered based on this flag
# From runtime.py:289-302
def _available_actions(cursor, workflow_version_id, from_step_definition_id, allow_revert: bool):
    cursor.execute(
        """SELECT DISTINCT action_type FROM workflow_transition_definition
           WHERE workflow_version_id = %s AND from_step_definition_id = %s""",
        (workflow_version_id, from_step_definition_id)
    )
    actions = [row["action_type"] for row in cursor.fetchall()]
    if not allow_revert:
        actions = [action for action in actions if action != "revert"]
    return actions
```

### Loop Guards

<ResponseField name="maxVisitsPerInstance" type="integer" optional>
  Maximum times this step can be visited in a single workflow instance. Required for steps in loops.
</ResponseField>

```python
# Visit count tracking in runtime.py:277-286
def _get_next_visit_count(cursor, workflow_instance_id, step_definition_id) -> int:
    cursor.execute(
        """SELECT COUNT(*) AS count FROM step_instance
           WHERE workflow_instance_id = %s AND step_definition_id = %s""",
        (workflow_instance_id, step_definition_id)
    )
    return cursor.fetchone()["count"] + 1
```

<Warning>
Steps in cycles must have `maxVisitsPerInstance` set to prevent infinite loops. The validator rejects cycles without visit guards.
</Warning>

## Step Instances

Each time a workflow enters a step, a step instance is created. This allows tracking revisits (after revert) and multiple attempts.

### Runtime Schema

```sql
CREATE TABLE step_instance (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    workflow_instance_id uuid NOT NULL REFERENCES workflow_instance(id),
    step_definition_id uuid NOT NULL REFERENCES workflow_step_definition(id),
    attempt_no integer NOT NULL DEFAULT 1,
    visit_count integer NOT NULL DEFAULT 1,
    status text NOT NULL DEFAULT 'pending',
    entered_at timestamptz NOT NULL DEFAULT now(),
    started_at timestamptz,
    completed_at timestamptz,
    waiting_since timestamptz,
    actor_user_id text REFERENCES "user"(id),
    result_action text,
    result_payload jsonb,
    error_payload jsonb
);
```

### Step Instance Status

```python
StepStatus = Literal[
    "pending",     # Created but not started
    "active",      # Currently processing
    "waiting",     # Paused on human task or subworkflow
    "completed",   # Successfully finished
    "rejected",    # Terminated via rejection
    "reverted",    # Reverted to previous step
    "failed",      # Failed due to error
    "cancelled"    # Manually cancelled
]
```

### Creating Step Instances

Step instances are created when the workflow engine enters a step:

```python
# From runtime.py:855-883
def _enter_step(cursor, workflow_instance, step_definition, trigger_action: str = "approve"):
    visit_count = _get_next_visit_count(cursor, workflow_instance["id"], step_definition["id"])

    cursor.execute(
        """INSERT INTO step_instance (
            workflow_instance_id,
            step_definition_id,
            attempt_no,
            visit_count,
            status,
            entered_at,
            started_at
        ) VALUES (%s, %s, 1, %s, 'active', now(), now())
        RETURNING *""",
        (workflow_instance["id"], step_definition["id"], visit_count)
    )
    step_instance = cursor.fetchone()
    
    # Update workflow pointer to current step
    cursor.execute(
        """UPDATE workflow_instance
           SET current_step_instance_id = %s, updated_at = now()
           WHERE id = %s""",
        (step_instance["id"], workflow_instance["id"])
    )
```

## Assignment Policy

Each human_task step has an assignment policy that controls how tasks are distributed and approved.

### Policy Schema

```sql
CREATE TABLE workflow_step_assignment_policy (
    id uuid PRIMARY KEY,
    step_definition_id uuid NOT NULL UNIQUE REFERENCES workflow_step_definition(id),
    approval_mode text NOT NULL DEFAULT 'priority_chain',
    required_approvals_count integer,
    priority_escalation_enabled boolean NOT NULL DEFAULT false,
    escalation_timeout_seconds integer,
    reminder_interval_seconds integer,
    max_escalation_count integer,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

### Approval Modes

```python
ApprovalMode = Literal[
    "priority_chain",   # Notify highest priority first, escalate on timeout
    "approve_any_one",  # First approval completes the step
    "approve_all",      # All assignees must approve
    "notify_all"        # Notify everyone, policy determines completion
]
```

<AccordionGroup>
  <Accordion title="priority_chain">
    Tasks are created in priority order. Only the highest priority task is `open`, others are `queued`. When a task is completed or times out, the next priority task opens.
    
    **Use case**: Sequential approval chain (requester → manager → director → CFO)
  </Accordion>
  
  <Accordion title="approve_any_one">
    All tasks are created as `open`. The first approval completes the step, remaining tasks are cancelled.
    
    **Use case**: Any team member can approve (support ticket assignment)
  </Accordion>
  
  <Accordion title="approve_all">
    All tasks are created as `open`. Every assignee must approve before the step completes.
    
    **Use case**: Multi-signature approval (compliance review requiring legal + security)
  </Accordion>
  
  <Accordion title="notify_all">
    All assignees are notified, but the policy's `required_approvals_count` determines completion.
    
    **Use case**: Quorum voting (3 of 5 board members must approve)
  </Accordion>
</AccordionGroup>

### Policy Example

```python

    reminderIntervalSeconds: int | None = None
```

```python
for step in payload.steps:
```
