## Overview

Workflows are reusable process templates that define a series of steps and transitions for business automation. The platform separates **static workflow definitions** from **runtime execution instances**, ensuring that changes to definitions don't corrupt in-flight workflows.

## Workflow Definition

A workflow definition is the top-level identity for a business process. Each definition can have multiple versions, allowing you to evolve workflows over time while maintaining backward compatibility.

### Core Properties

<CodeGroup>

## Workflow Versions

Each workflow definition maintains immutable versions. Runtime instances reference specific versions, not the mutable parent definition.

### Version Schema

```sql
CREATE TABLE workflow_definition_version (
    id uuid PRIMARY KEY,
    workflow_definition_id uuid NOT NULL REFERENCES workflow_definition(id),
    version_no integer NOT NULL,
    is_published boolean NOT NULL DEFAULT false,
    version_label text,
    definition_snapshot jsonb NOT NULL,
    graph_json jsonb NOT NULL,
    builder_layout jsonb NOT NULL,
    created_by text REFERENCES "user"(id),
    created_at timestamptz NOT NULL DEFAULT now(),
    UNIQUE (workflow_definition_id, version_no)
);
```

### Version Properties

- **version_no**: Incremental version number (1, 2, 3...)
- **is_published**: Only one version can be published at a time
- **definition_snapshot**: Complete denormalized export for auditing
- **graph_json**: Canonical JSON generated from steps and transitions
- **builder_layout**: React Flow visual editor state (nodes, edges, viewport)

<Warning>
Only published versions can be used to start new workflow instances. Unpublished versions are drafts.
</Warning>

## Workflow Instance

A workflow instance represents a single execution of a workflow definition. Instances track their current state, step, and history.

### Runtime Schema

```sql
CREATE TABLE workflow_instance (
    id uuid PRIMARY KEY,
    workflow_version_id uuid NOT NULL REFERENCES workflow_definition_version(id),
    business_key text,
    run_number integer NOT NULL DEFAULT 1,
    status text NOT NULL DEFAULT 'running',
    current_step_instance_id uuid,
    started_by text REFERENCES "user"(id),
    started_at timestamptz NOT NULL DEFAULT now(),
    completed_at timestamptz,
    parent_workflow_instance_id uuid,
    parent_step_instance_id uuid,
    lock_version integer NOT NULL DEFAULT 0,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now()
);
```

## Workflow Validation

Before a workflow version can be published, the engine validates the graph structure to prevent invalid configurations.

### Validation Rules

The validation logic is implemented in `apps/api/src/api/workflows.py:72-192`:

1. **Exactly one start step** is required
2. **At least one terminal step** (end step or `isTerminal=true`)
3. **Unique step codes** within a version
4. **All non-terminal steps** must have outgoing transitions
5. **All steps must be reachable** from the start step
6. **Transition targets must exist**
7. **No unconditional self-loops**
8. **Cycles require guards**: Loops must have `maxVisitsPerInstance` or conditional exits

### Validation Example

```python
# From workflows.py:156-183
def dfs(step_code: str) -> None:
    visited.add(step_code)
    active_lookup.add(step_code)
    active_stack.append(step_code)

    for transition in outgoing.get(step_code, []):
        next_step = transition.toStepCode
        if not next_step:
            continue

        if next_step in active_lookup:
            cycle = active_stack[active_stack.index(next_step):]
            if not any(
                steps_by_code[item].maxVisitsPerInstance is not None 
                for item in cycle
            ):
                errors.append(
                    "Detected a cycle without a maxVisitsPerInstance guard on steps: "
                    + ", ".join(cycle)
                )
            continue

        if next_step not in visited:
            dfs(next_step)
```

<Note>
The validator doesn't attempt to prove all conditional loops are safe—it enforces structural rules to prevent obviously unsafe infinite workflows.
</Note>

### Workflow Instance Data

Workflow variables are stored separately to avoid bloating the main instance row:

```sql
CREATE TABLE workflow_instance_data (
    workflow_instance_id uuid PRIMARY KEY,
    input_data jsonb NOT NULL DEFAULT '{}'::jsonb,
    context_data jsonb NOT NULL DEFAULT '{}'::jsonb,
    output_data jsonb NOT NULL DEFAULT '{}'::jsonb,
    last_error jsonb,
    updated_at timestamptz NOT NULL DEFAULT now()
);
```

- **input_data**: Original request payload (immutable)
- **context_data**: Mutable state bag updated as workflow progresses
- **output_data**: Final result when workflow completes
- **last_error**: Error details if workflow fails

## Workflow History and Audit

All workflow state changes are tracked in the status history table:

```sql
CREATE TABLE workflow_status_history (
    id uuid PRIMARY KEY,
    workflow_instance_id uuid NOT NULL,
    old_status text,
    new_status text NOT NULL,
    reason text,
    changed_by_action_id uuid,
    created_at timestamptz NOT NULL DEFAULT now()
);
```

Status changes are recorded in `runtime.py:59-79`:

```python
def _record_status_history(
    cursor,
    workflow_instance_id,
    old_status: str | None,
    new_status: str,
    reason: str | None = None,
    changed_by_action_id=None,
) -> None:
    cursor.execute(
        """INSERT INTO workflow_status_history (...) VALUES (...)""",
        (workflow_instance_id, old_status, new_status, reason, changed_by_action_id)
    )
```
