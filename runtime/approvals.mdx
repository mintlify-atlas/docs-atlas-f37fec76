## Overview

Human tasks must be actioned by assigned users to advance the workflow. The platform supports approve, reject, revert, and custom actions with optional remarks and payload data.

## Generic Task Action Endpoint

### POST /api/v1/human-tasks/:task_id/actions

Submit any action type to a task with full control over action parameters.

**Request Body:**

```json
{
  "actionType": "approve",
  "remark": "Budget approved by finance team",
  "payload": {
    "approvedAmount": 15000,
    "notes": "Vendor verification completed"
  },
  "actionCode": null
}
```

<ParamField path="actionType" type="string" required>
  The action to perform: `"approve"`, `"reject"`, `"revert"`, or `"custom"`
</ParamField>

<ParamField path="remark" type="string" optional>
  Optional comment explaining the decision. Required if the step has `remark_required_on_approve`, `remark_required_on_reject`, or `remark_required_on_revert` enabled.
</ParamField>

<ParamField path="payload" type="object" default="{}">
  Additional data to attach to the action. This is stored in the `workflow_action` table and can be used by output mappings.
</ParamField>

<ParamField path="actionCode" type="string" optional>
  For custom actions, the specific transition code to follow (matches `action_code` on a `workflow_transition_definition`).
</ParamField>

**Response (200 OK):**

```json
{
  "workflowInstanceId": "workflow-instance-uuid",
  "stepInstanceId": "step-instance-uuid",
  "actionType": "approve",
  "workflowStatus": "waiting",
  "nextStepCode": "cfo-approval"
}
```

<ResponseField name="workflowInstanceId" type="UUID">
  The workflow instance that was updated
</ResponseField>

<ResponseField name="stepInstanceId" type="UUID">
  The step instance that was completed
</ResponseField>

<ResponseField name="actionType" type="string">
  The action that was applied
</ResponseField>

<ResponseField name="workflowStatus" type="string">
  The new workflow status: `"running"`, `"waiting"`, `"completed"`, or `"rejected"`
</ResponseField>

<ResponseField name="nextStepCode" type="string | null">
  The step code the workflow advanced to, or `null` if the workflow completed
</ResponseField>

## Specialized Action Endpoints

For convenience, the platform provides action-specific endpoints that set `actionType` automatically.

### Approve Action

**POST /api/v1/human-tasks/:task_id/approve**

```json
{
  "remark": "Looks good",
  "payload": {}
}
```

Equivalent to calling the generic endpoint with `"actionType": "approve"`.

### Reject Action

**POST /api/v1/human-tasks/:task_id/reject**

```json
{
  "remark": "Insufficient documentation",
  "payload": {
    "reason": "missing-vendor-invoice"
  }
}
```

Equivalent to calling the generic endpoint with `"actionType": "reject"`.

### Revert Action

**POST /api/v1/human-tasks/:task_id/revert**

```json
{
  "remark": "Send back to requester for corrections",
  "payload": {}
}
```

Equivalent to calling the generic endpoint with `"actionType": "revert"`.

```typescript Frontend Action Submission
// From runtime-console.tsx:361-399
async function submitTaskAction(task: HumanTask, actionType: TaskActionType) {
  setIsBusy(true)
  setError(null)
  setStatus(`Submitting ${actionType} action for ${task.stepLabel}...`)

  try {
    const response = await authedFetch(`/api/v1/human-tasks/${task.id}/actions`, {
      method: "POST",
      body: JSON.stringify({
        actionType,
        remark: taskRemarks[task.id] || null,
        payload: {
          source: "runtime-console",
        },
      }),
    })

    const payload = (await response.json()) as {
      workflowInstanceId?: string
      nextStepCode?: string | null
      workflowStatus?: string
      detail?: string
    }
    if (!response.ok) {
      throw new Error(payload.detail ?? "Unable to submit task action.")
    }

    setStatus(
      `Task action ${actionType} recorded. Workflow is now ${payload.workflowStatus ?? "updated"}.`,
    )
    setTaskRemarks((current) => ({ ...current, [task.id]: "" }))
    await Promise.all([refreshAll(), loadInstanceDetail(task.workflowInstanceId)])
  } catch (actionError) {
    setError(actionError instanceof Error ? actionError.message : "Unable to action task.")
    setStatus("Task action failed.")
  } finally {
    setIsBusy(false)
  }
}
```

## Action Processing Logic

When a task action is submitted, the backend executes the following steps:

### 1. Validate Task State


Only tasks with `status = "open"` can be actioned.

### 2. Verify User Assignment


### 3. Check Remark Requirements

### 6. Handle Approval Mode Logic

The platform checks the `approval_mode_snapshot` to determine whether to advance the workflow immediately or wait for more tasks.

#### approve_all Mode

```python
# From runtime.py:1343-1364
if approval_mode == "approve_all":
    if action_type == "approve":
        cursor.execute(
            """
            SELECT COUNT(*) AS remaining
            FROM human_task
            WHERE step_instance_id = %s AND status IN ('open', 'queued', 'claimed') AND id <> %s
            """,
            (task_row["step_instance_id"], task_row["id"]),
        )
        remaining = cursor.fetchone()["remaining"]
        if remaining > 0:
            advance = False
            cursor.execute(
                """
                UPDATE step_instance
                SET result_action = 'approve'
                WHERE id = %s
                """,
                (task_row["step_instance_id"],),
            )
```

If any tasks remain open, the workflow does not advance yet.

#### approve_any_one Mode

```python
# From runtime.py:1366-1378
elif approval_mode == "approve_any_one":

            """
            SELECT COUNT(*) AS remaining
```

## Error Handling

**403 Forbidden:**

```json
{
  "detail": "You are not assigned to this task."
}
```

The authenticated user does not match the task assignment.

**404 Not Found:**

```json
{
  "detail": "Task not found."
}
```

The `task_id` does not exist.

**409 Conflict:**

```json
{
  "detail": "Only open tasks can be actioned."
}
```

The task status is not `"open"`.

**422 Unprocessable Entity:**

```json
{
  "detail": "A remark is required for this action."
}
```

The step requires a remark but none was provided.

```json
{
  "detail": "No matching transition exists for this action."
}
```

The action type has no defined transition in the workflow graph.

## Workflow Advancement

After task completion, the workflow can:

1. **Remain waiting:** If approval mode requires more tasks to complete
2. **Advance to next step:** If a matching transition exists
3. **Complete:** If the transition points to `null` (terminal)
4. **Reject:** If the action is `"reject"` and no next step is defined

The `nextStepCode` in the response indicates the result:

- `"manager-review"`: Workflow advanced to the manager review step
- `null`: Workflow reached a terminal state (completed or rejected)
