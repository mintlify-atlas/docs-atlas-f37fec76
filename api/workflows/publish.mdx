---
title: Publish Workflow Definition
api: POST /api/v1/workflow-definitions/{definition_id}/publish
---

Publishes the latest version of a workflow definition, making it the active version that will be used for new workflow instances. This also changes the workflow definition status to `active`.

## Authentication

Requires Bearer token authentication.

## Path Parameters

<ParamField path="definition_id" type="UUID" required>
  The unique identifier of the workflow definition to publish.
</ParamField>

## Response

## Behavior

### Version Publishing

- Only the **latest version** is published by this operation
- All previous versions are automatically unpublished (their `is_published` flag is set to `false`)
- Only one version can be published at a time per workflow definition

### Status Change

- The workflow definition's status changes from `draft` to `active`
- If already `active`, the status remains `active` but the published version changes

### New Instances

- After publishing, all new workflow instances created from this definition will use the newly published version
- Existing running instances continue using their original version (no impact)

### Validation

- Publishing does not re-run validation; the version was already validated when created
- Ensure the workflow has been tested before publishing to production

## Workflow Lifecycle

### 1. Create

```bash
POST /api/v1/workflow-definitions
# Creates definition with version 1 in draft state
# Status: draft
```

### 2. Update (Optional)

```bash
PUT /api/v1/workflow-definitions/{id}
# Creates version 2 in draft state
# Status: still draft
```

### 3. Publish

```bash
POST /api/v1/workflow-definitions/{id}/publish
# Publishes latest version (version 2)
# Status: active
```

### 4. Use

```bash
POST /api/v1/workflow-instances
# New instances use the published version (version 2)
```

### 5. Iterate

```bash
PUT /api/v1/workflow-definitions/{id}
# Creates version 3 in draft state
# Published version remains version 2
# Status: active (unchanged)

POST /api/v1/workflow-definitions/{id}/publish
# Publishes version 3, unpublishes version 2
# New instances now use version 3
```

## Best Practices

### Testing Before Publishing

1. Create a test workflow instance using the draft version
2. Validate the workflow behaves as expected
3. Check all transitions, assignments, and notifications
4. Only publish after successful testing

### Version Control Strategy

- **Minor changes**: Create new version, test in draft, publish when ready
- **Major changes**: Consider creating a new workflow definition with a different key
- **Breaking changes**: Always create a new version; never modify published versions

### Rollback Strategy

There is no built-in rollback mechanism. To revert to a previous version:

1. Retrieve the previous version's definition
2. Create a new version with that definition (PUT)
3. Publish the new version

This creates a new version number with the old definition, preserving audit trail.

## Use Cases

1. **Production Deployment**: Promote a tested workflow to production
2. **Version Switching**: Switch between different workflow versions
3. **Emergency Updates**: Quickly publish critical fixes
4. **Workflow Activation**: Make a new workflow available for use
