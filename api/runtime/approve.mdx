---
title: Approve Task
api: POST /api/v1/human-tasks/{task_id}/approve
---

Approves a human task, advancing the workflow to the next step or completing the workflow if this is the final approval step.

## Authentication

Requires Bearer token authentication. The authenticated user must be assigned to the task.

## Path Parameters

<ParamField path="task_id" type="UUID" required>
  The unique identifier of the task to approve.
</ParamField>

## Request Body

<ParamField body="remark" type="string">
  Optional comment or explanation for the approval. Required if the step definition has `remarkRequiredOnApprove` set to true.
</ParamField>

<ParamField body="payload" type="object" default="{}">
  Optional additional data to include with the approval action. This data becomes available in the workflow context.
</ParamField>

<ParamField body="actionCode" type="string">
  Optional custom action code (used for custom action types). Not typically needed for approve actions.
</ParamField>

## Response

<ResponseField name="workflowInstanceId" type="UUID">
  The ID of the workflow instance.
</ResponseField>

<ResponseField name="stepInstanceId" type="UUID">
  The ID of the step instance that was completed.
</ResponseField>

<ResponseField name="actionType" type="string">
  The type of action performed (`approve`).
</ResponseField>

<ResponseField name="workflowStatus" type="string">
  The new status of the workflow after this action:
  - `running` - Workflow advanced to next step
  - `waiting` - Still waiting for other approvals (in approve_all mode)
  - `completed` - Workflow completed successfully
  - `paused` - Workflow paused (e.g., waiting for subworkflow)
</ResponseField>

<ResponseField name="nextStepCode" type="string | null">
  The code of the next step the workflow advanced to, or null if workflow completed.
</ResponseField>

## Status Codes

<ResponseField name="200" type="OK">
  Task approved successfully.
</ResponseField>

<ResponseField name="401" type="Unauthorized">
  Missing or invalid authentication token.
</ResponseField>

<ResponseField name="403" type="Forbidden">
  User is not assigned to this task.
</ResponseField>

<ResponseField name="404" type="Not Found">
  Task not found.
</ResponseField>

<ResponseField name="409" type="Conflict">
  Task is not in an actionable state (e.g., already completed).
</ResponseField>

<ResponseField name="422" type="Unprocessable Entity">
  Request validation failed:
  - Missing required remark when `remarkRequiredOnApprove` is true
  - No matching transition exists for the approval action
</ResponseField>

## Approval Modes

### Priority Chain

In `priority_chain` mode, approvals happen sequentially:
- Only one task is `open` at a time
- Approval advances to the next task in the chain
- Workflow continues when all chain approvals complete

### Approve Any One

In `approve_any_one` mode:
- Multiple tasks are `open` simultaneously
- First approval completes the step
- Other tasks are automatically cancelled

### Approve All

In `approve_all` mode:
- All assigned tasks must be approved
- Each approval keeps workflow in `waiting` status
- Last approval advances the workflow
- Response includes `workflowStatus: "waiting"` until all approve

## Behavior

### Workflow Advancement

When a task is approved:
1. Task status changes to `completed`
2. Action is recorded in workflow history
3. Step instance is marked as completed
4. Workflow engine evaluates transitions from current step
5. If transition exists, workflow advances to next step
6. If no transition or terminal step, workflow completes

### Context Updates

Any data in the `payload` field is merged into the workflow context and becomes available to subsequent steps:

```json
{
  "payload": {
    "approvedAmount": 1500.00,
    "approverLevel": "manager"
  }
}
```

### Notifications

When a task is approved:
- Real-time events are published to subscribers
- Next assignee receives notification (if applicable)
- Workflow status change is broadcast
